## What this is
A static, searchable reference app for Equestrian Australia (EA) dressage
equipment and rules. Single-page vanilla HTML/CSS/JS in `site/`; content
lives in `site/data.js` (a JS file, not fetched JSON). No framework, no
bundler, no backend — keep it that way unless the owner asks otherwise.

Current shape: **157 items**, four tabs — **Equipment** (searchable card
grid), **Equipment Rules** (rule topics + galleries), **Dressage Tests**
(sub-tabs: EA · Young Horse · FEI; EA Para is built but currently hidden
behind the `SHOW_PARA` flag in index.html), and **Dressage Rulebook** (link
to the official PDF). PWA: full-image offline precache, an in-app
"new version — tap to update" banner, a "Share this app" QR popup, and a
branded install guide (`site/install-guide.pdf`, regenerated by
`tools/make_install_guide.py`).

## Deployment (two hosts, one push)
Pushing to `main` deploys **both**: GitHub Pages (`agunn.github.io/ea-dressage`)
and Cloudflare (`ea-dressage.eq-au.workers.dev`, the share URL) via
Cloudflare's Git-connected Workers Build. Config is `wrangler.jsonc`
(assets-only worker serving `./site`) — not `wrangler.toml`, and no GitHub
Actions for Cloudflare. If the Cloudflare Git build lags or their API wobbles,
force it directly:
`CLOUDFLARE_API_TOKEN=$(cat ~/.cf-token) CLOUDFLARE_ACCOUNT_ID=45afe5b260e6420f0db6c85fe07ff11e npx wrangler deploy`
Owner publishes only on explicit request — edits wait on the NAS until then.

**Staging** (manual only, never via git push): `ea-dressage-staging2.eq-au.workers.dev`
— same account, deployed with the command above plus `--name ea-dressage-staging2`
(the original `-staging` worker was deleted after Chrome soured on its
half-set-up first minutes; index.html swaps to manifest-staging.json on
any hostname containing `-staging`, naming the install "EA Staging").
When the owner says "stage it", deploy the current NAS `site/` there (uncommitted
is fine; that's the point) for on-device testing before a real publish. Staging
installs on devices are separate apps from production ones.

## Golden rules
- `site/` must remain deployable as-is on any static host and keep working
  when opened directly from `file://` (data is a JS file, not fetched JSON —
  do not change that).
- This is a **rules reference**: content accuracy beats style. Never invent,
  reword or "improve" rule wording; when sources conflict, surface the
  conflict to the owner (see the `conflicts` array in data.js) rather than
  picking a side. Owner-approved wording changes are logged below.
- Do not regenerate `site/data.js` via the tools pipeline for small content
  edits — hand-edit data.js. The pipeline is only for new source-document
  versions, and `tools/build_merged_data.py` contains hand-curated content
  (rules topics, rulebook items, enrichments) that must be reviewed, not
  blindly rerun.
- Bump the cache name in `site/sw.js` (`ea-equipment-vNN`) whenever content
  or code changes, or returning users see stale cached versions. (Now v108+.)
- When publishing, also update the "Site last updated D Month YYYY" line in
  the footer of `site/index.html` to the publish date (hand-edited string).

## Data model (site/data.js -> EA_DATA)
- `items[]`: id, section (1–8), sectionName, title, description, notes,
  status (`permitted|prohibited|conditional`), penalty (`Elimination`,
  `Minus 0.5%`, or null), condition, scope (`competition|warm-up`),
  updated, images[], source (`annex|rules`), ruleRef.
- `rules[]`: prose rule topics {id, ref, title, body[], images[{src,cap}]}.
- `galleries[]`: numbered official illustrations (bits 5.20, nosebands 5.21).
- `policy`: EADC noseband policy page. `conflicts[]`: cross-document
  discrepancies shown as flagged cards (currently empty — throat lash
  resolved). `meta`: version strings shown in the header.

## UI structure (site/index.html)
Everything is in the one `index.html` (inline `<style>` + `<script>`). Tabs
are icon+label cells (bottom bar on phones/tablets, top row on desktop).
Equipment shows a status segmented control, scrollable/section chips, a
clickable "i" info circle that doubles as the offline-download progress ring,
and a card grid. Item, rule, policy, advisory, install and share views all
render through one shared `<dialog>` modal. Rule-ref tags (cards and item
popup) link to the rule text. Styling via CSS custom properties; verdict
colours: green permitted, red prohibited, amber conditional; brand navy
`#002942`.

## Testing
No test suite. Sanity-check with Node after data edits:
    node -e "const fs=require('fs');(0,eval)(fs.readFileSync('site/data.js','utf8').replace('const EA_DATA','globalThis.EA_DATA'));console.log(EA_DATA.items.length)"
Verify every `images[]` entry exists in `site/images/`, and syntax-check the
inline scripts. Keep total site size small (~3 MB) — images pre-compressed to
≤640–700px JPEG q≈83. Owner reviews visually on his own devices; don't rely
on the preview pane (it can't render the logo images).
